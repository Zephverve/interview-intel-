## P1-1 · 为什么用 LangGraph？缺点？什么时候过度设计？ `[面经原文]`

📊 出现 7 次 · 美团/高德/阿里 · 八股/工程

📖 核心要点：① 循环/人机确认/checkpoint ② 简单链路用 ReAct ③ 先跑通再上图

🗣️ **标准口语答案**

"我选 LangGraph 的场景是：流程 **不是一条直线**——有多分支、需要人机确认、或者任务半路断了要能 **checkpoint 恢复**。它把 state 和边写清楚，比 while 里塞一堆 if-else 好维护，调试时也能看到执行路径。

缺点是团队要接受 **图的心智模型**，小改动有时要动 reducer 和边，**样板代码偏多**。如果业务只是『分类 → 单次检索 → 回答』，用简单 ReAct 循环就够了，上图确实偏重。

我判断有没有过度设计看三点：有没有真实的 **中断恢复需求**；流程会不会 **频繁改**；团队能不能 hold 住这套抽象。我自己的科研问答里用 LangGraph 是因为有 **查询改写→检索→重排→引用校验→失败重试** 的多节点流程，平均 1.3 轮校验才通过，这种状态机比裸 ReAct 清晰。

**先 Workflow/ReAct 跑通，遇到循环/回溯痛点再迁 LangGraph**——别为了炫技提前上图。"

🔍 追问：checkpoint 存哪？非幂等 tool 中断点怎么设？

## P1-2 · RAG 怎么优化？知识库错误或过期怎么治理？ `[面经原文]`

📊 出现 9 次 · Minimax/美团/科研岗 · RAG

📖 核心要点：① 混合检索+重排 ② 过期/错误治理 ③ 验证节点

🗣️ **标准口语答案**

"RAG 优化我按 **检索→排序→生成** 三层做。

检索层：**BM25 + 向量混合**，解决专有名词和缩写；**query 改写**处理口语化输入；**分层 chunk**——我用的 512 子块 + 2k 父块，Recall@5 从 0.51 提到 0.78。

排序层：**Cross-Encoder 重排**，性价比最高的提升之一。

生成层：**引用校验**——生成结果必须能对应到检索块，校验失败自动重试，平均 1.3 轮通过；引用通过率 86%。

**知识过期/错误**是 Minimax 面经高频 [2]。我的做法是：文档带 **版本号和时间戳**；高频知识 **TTL**；写入前 **来源分级**（论文库 > 用户上传）；生成前加 **验证节点**——『当前块是否足够回答？来源是否可信？』；对关键域 **人工审核队列**。线上还要 **badcase 回流**改 chunk 或改文档，不能只改 prompt。"

🔍 追问：GraphRAG 用过吗？多跳推理怎么做？

## P1-3 · Multi-Agent 怎么通信？决策冲突怎么解决？ `[面经原文]`

📊 出现 6 次 · Minimax/字节 · Multi-Agent

📖 核心要点：① RPC/黑板/消息队列 ② 冲突：规则/评审 Agent ③ 何时不用 Multi-Agent

🗣️ **标准口语答案**

"Multi-Agent 通信常见三种：**编排器 RPC**——主 Agent 下发子任务收结果，像函数调用；**共享 checkpoint/黑板**——约定 schema 读写；**消息队列**——解耦但调试难。无论哪种都要设 **超时、子任务失败冒泡、避免双写乱 state**。

**决策冲突** MiniMax 面经很典型 [2]——比如设计 Agent 和代码 Agent 结论不一致。解法：① **技术方案驱动**——PRD 先定约束，Agent 在约束内发挥；② **优先级规则**——安全/合规 > 性能 > 美观；③ **评审 Agent 或人工 gate** 做仲裁；④ 固定流水线里 **不让并行 Agent 改同一资源**。

什么时候 **不** 用 Multi-Agent？任务固定、参数稳定——**封装成 Skill** 更省协调成本。Multi-Agent 的价值是 **独立 context、不同 retry 策略、并行角色**，不是为了炫技。"

🔍 追问：为什么不用 Skill 封装？A2A 了解吗？

## P1-4 · Harness Engineering 是什么？ `[推断]`

📊 近 3 个月热点 · 字节/知乎 · 工程

📖 核心要点：① 95% 是脚手架 ② 校验/沙箱/trace/评测 ③ 驯服不确定性

🗣️ **标准口语答案**

"**Harness** 是让 Agent 在生产里 **可控、可观测、可恢复** 的外层工程，不是 prompt 写得多妙。

模型调用可能只占 5%，其余是：**Pydantic 强校验** tool 输出；**沙箱**跑 code_run；**checkpoint** 断点续传；全链路 **trace**；**RAGAS/DeepEval** 自动评测；**Fail-Closed** 权限——工具删文件要白名单。

字节一面直接问 harness [13]。我和 GenericAgent 对齐的理解是：ReAct 循环是『大脑』，Harness 是『神经系统+安全网』——max_turns、重复检测、空响应退出、Mixin 故障转移都属于 Harness。

面试金句：**大厂缺的不是会调 API 的人，是用后端思维约束模型不确定性的人**。"

🔍 追问：和 OpenClaw 关系？CLI vs IDE Agent？

## P1-5 · Agent 怎么评测？ `[面经原文]`

📊 出现 6 次 · 百度/美团 · 评测

📖 核心要点：① 离线+在线 ② Evaluation 独立 ③ 指标：完成率/tool 准确率

🗣️ **标准口语答案**

"Agent 评测我分 **离线** 和 **在线**。

离线：**自建分层用例**——成功路径 + 故意失败路径（工具超时、检索为空）；脱敏真实日志；工具在 **沙箱** 跑，不污染生产。我维护 **120 问黄金集**，按 session 回溯失败样本。

指标：**任务完成率**、**tool 调用准确率**、平均步数、**P95 延迟**、用户 **显式纠正率**。

在线：A/B 或灰度桶对比；全链路 trace 每步打标，badcase 进池子改 prompt/文档/工具。

百度二面强调 **Evaluation 模块要独立** [3]——它是审判系统的，不能和在线链路耦在一起。工具用 **RAGAS/DeepEval** 做回归，别只靠手点。"

🔍 追问：LLM-as-judge 偏差怎么控？

## P1-6 · 工具/Skill 过多，Router 失败怎么办？ `[推断]`

📊 美团二面热点 · 工具路由

📖 核心要点：① Top-K Skill routing ② 失败：扩大K/规则/ask_user ③ 工具描述质量

🗣️ **标准口语答案**

"工具或 Skill 一多，全塞进 prompt 会 **挤占 context、降低选型准确率**。

做法是 **两层路由**：第一层用规则、小分类模型或向量检索，从全量筛 **Top-K**（比如 3–5 个 Skill），只把这 K 个的 description 和 schema 注入；第二层让 LLM 做 Function Calling。

**Router 失败**时：① 扩大 K 重试；② **降级到关键词规则**；③ 让模型先输出『需要哪类能力』再二次路由；④ 仍不行 **ask_user** 澄清意图。

工程上每个 tool **单一职责 + 清晰描述 + 示例**，比堆数量重要。GenericAgent 用 **9 个原子工具 + code_run 图灵完备扩展**，而不是注册 100 个窄工具——这是另一种降复杂度思路。"

🔍 追问：Tool RAG 是什么？100+ 工具场景？

## P2-1 · Harness vs Prompt Engineering `[推断]`

📊 字节/知乎 2026

🗣️ **标准口语答案**

"**Prompt Engineering** 优化单次怎么问——system prompt、few-shot、输出格式，是 **这一轮怎么说话**。

## P2-4 · Agentic CPT → SFT → RL（算法向）`[面经原文]`

📊 字节算法向

🗣️ **标准口语答案**

"**Agentic CPT** 用 Agent 轨迹数据做继续预训练，让模型熟悉 multi-step 工具格式。**SFT** 在指令数据上微调，常 **mask observation token**，只学模型的 reasoning 和 tool call 格式。**RL/GRPO** 用任务成功率或人工偏好优化长期策略，适合 tool 选错、步骤冗余。

应用岗不必推导 PPO 损失，但要能说：**SFT 解决格式和风格，RL 解决长期奖励，RAG 解决知识**——分工不同。我做过 **QLoRA SFT** 科研问答，1.8k 条指令，loss 0.42，解决的是『基于引用作答』风格，不是 tool 路由。"

🔍 追问：DPO 和 PPO 选型？

## P2-5 · AI 出码留用率低，问题在哪？diff 控制？ `[面经原文]`

📊 Minimax [2]

🗣️ **标准口语答案**

"Minimax 面经提到留用率 50%+，剩下 50% 常见问题是：**逻辑看起来对其实错**、**改范围过大**、**没跑测试**、**不了解存量项目结构**。

治理：**diff 级控制**——限制 AI 只能改指定目录/文件；**强制跑测试/Playwright** 再合入；**知识库 .rules** 注入项目规范；**技术方案先行**再出码，不让 AI 直接从 PRD 跳到代码。复杂业务场景质量下降，往往是 **context 里缺架构约束** 或 **知识过期**。"
